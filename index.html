The problem regulators actually find.

Across recent enforcement orders, regulators rarely allege the bank missed a typology. They allege the bank cannot evidence what it did. Process-pain research across 10 primary-source documents — the root cause in plain language.

PAIN-1

The program can’t evidence what it did

When the examiner walks in, months of alert reviews, rule changes, and case closures exist only in Excel. The audit ledger needs to be on the shelf before the exam starts.

PAIN-5

Data arrives late, dirty, and inconsistently joined

The binding constraint behind most consent orders is the data layer — not detection logic. Eleven distinct failure modes documented in the data-problem whitepaper, all traceable to enforcement language.

PAIN-9

Rule changes can’t be validated without a rebuild

When a rule is tuned, nobody can prove the prior version was correct without replaying it. The backtester closes this loop: same spec, same data, identical output hashes.

What we built — one Manifest, four layers.

Policy, data contracts, detection rules, case workflow and regulator mapping live in one versioned document — the Compliance Manifest (an aml.yaml file, for engineers). Every runtime artifact is generated from it; an immutable hash-chained audit ledger verifies it.

What this is — and what it isn’t.

This is the spec / control-plane layer above your AML stack — not a replacement for any tool inside it. Think Terraform for AML controls.

✓ What it does

  • Declares rules, data contracts, case workflow, and metrics in one versioned YAML
  • Generates SQL detectors, MRM dossiers, SAR narratives, and exam ZIPs from that YAML
  • Produces a SHA-256 hash-chained audit ledger an examiner can replay byte-for-byte
  • Runs inside your perimeter — no data leaves, no per-event metering
  • Governs whatever detection engine, data pipeline, and case tool you already own

✕ What it isn’t

  • Not a transaction-monitoring platform (Actimize, SAS AML, Quantexa)
  • Not a data warehouse or pipeline (Snowflake, Databricks, dbt)
  • Not a case-management system (Pega, Hummingbird, Unit21)
  • Not a SAR/STR filing service
  • Not an AI detection model — though rules can call ML scorers via python_ref
In-bank, not SaaS

No data egress. Apache 2.0. Runs in your perimeter.

The dominant AML AI-vendor model ships your data to a cloud. This framework runs inside your VPC, on-prem cluster, or air-gapped environment. No per-event meter, no perpetual data-export contract. SR 11-7 / E-23 / SS1/23 model-risk evidence stays in-house.

SR 26-2 ready

Model inventory entries generated per scenario, not per system.

SR 26-2 (effective Apr 2026) unified AML and credit model governance. The Compliance Manifest auto-generates SR-26-2-aligned model documentation — purpose statement, conceptual soundness, ongoing monitoring, outcome analysis — for every TM rule that scores or categorises customers. This is the only point on the chart that’s both audit-mature and vendor-unlocked.
